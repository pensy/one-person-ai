# Gitee Sync

本仓库配置了两个远程，保持双向镜像同步：
- `gitee` — Gitee（国内镜像）
- `origin` — GitHub

**每次完成一个代码修改请求后，在回复用户之前运行此技能。** 不要等用户说"提交"——默认就要同步。用户若不想推送，会主动阻止。

一个任务 = 一次提交。不要把一次请求拆成多个细碎提交。

## 工作流程

### 1. 检查是否有变更

运行 `git status --short`。

- 输出为空 → 工作区干净，无需操作。直接告知用户"无变更需要同步"并结束。
- 有输出 → 进入下一步。

### 2. 生成提交信息

运行 `git diff` 与 `git diff --staged` 查看改动，必要时 `cat` 新增文件了解内容。然后按下方规范生成**一条**中文提交信息。

信息只生成在脑子里/回复里，不要自己 `git commit`——脚本会负责提交。

### 3. 执行提交与双推

调用脚本，把提交信息作为唯一参数传入：

```bash
bash .agents/skills/gitee-sync/scripts/sync.sh "feat: 你的提交信息"
```

脚本内部会：暂存全部变更（含新增/删除）→ 提交 → 推送当前分支到 gitee 和 origin，并逐个报告结果。

### 4. 报告结果

向用户简述：提交了什么、两个远程各自的推送状态（成功/失败）。如果某个远程推送失败，把失败原因一并转达。

## 提交信息规范

格式：`<type>: <中文描述>`

**type 速查**（以本次变更的主要目的为准）：

| type | 用于 |
|---|---|
| `feat` | 新增功能、新文件、新服务 |
| `fix` | 修复 bug、错误、构建失败 |
| `refactor` | 重构，不改变外部行为 |
| `perf` | 性能优化 |
| `docs` | 文档变更（README、注释说明等） |
| `style` | 代码格式调整，不影响逻辑 |
| `test` | 测试相关 |
| `chore` | 构建、依赖、配置、Dockerfile、脚手架 |

**描述要求**：
- 中文，简洁，说清"做了什么"
- 多个改动用 `，` 或 ` — ` 分隔
- 不要写"update files"这种无意义信息

**贴合本项目历史的例子**（真实提交）：
- `feat: 启用前端服务，完善 Dockerfile`
- `fix: 修复 frontend 构建 — 降级 pnpm 到 v9，修复 workspace 配置`
- `fix: 统一 Dockerfile 位置，修复构建路径`
- `chore: 升级依赖版本`

## 边界情况

- **当前不在 main 分支**：照常推送当前分支即可，脚本会自动取当前分支名。不要擅自切分支。
- **push 被拒绝（non-fast-forward）**：说明远程有本地没有的提交。不要自动 `--force`。告知用户存在冲突，询问是否 `git pull --rebase` 后重试。
- **只有 .agents/skills 自身的改动**：仍按正常流程提交同步。
- **用户明确说"先别推"/"只改不提交"**：尊重用户，本次跳过。
